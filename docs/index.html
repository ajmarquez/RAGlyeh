Lovecraft RAG Engine — Milestone roadmap (beginner-friendly)

A compressed, motivation-first learning plan: start from zero with FastAPI, get to a real RAG answer early, then slow down where the harder work really begins: persistence, embeddings, evaluation, and tracing.

Shape: 6 milestones (~10-12 weeks at 4h/week) Pacing: early wins in hours, later phases intentionally slower Output: reusable Lovecraft-to-any-docset RAG template
What you will build: A FastAPI RAG service over public-domain Lovecraft text: story ingestion, chunking, keyword retrieval, answer generation with citations, then Postgres, pgvector, evals, traces, and a small demo UI. What this teaches: API basics first, then retrieval, then the more realistic engineering work around persistence, embeddings, evaluation, and debugging. How to use this roadmap: Work through milestone checklists in order. Milestones are not equal effort; early milestones are short, later ones require more debugging and validation.
Tip: your first visible wins should happen fast. Later milestones slow down because persistence, embeddings, and evaluation are more complex than basic API work.
Milestone A Foundations 0%

Goal

Get a clean local FastAPI project running from zero.

Estimated total

4-6 hours (often faster if focused).

What you will learn

  • How uvicorn app.main:app --reload maps to your app entrypoint.
  • How to keep dependencies reproducible and project layout clear.

Checklist

Definition of done

Done when /health runs locally and dependencies are reproducible on another machine.

Loose pacing

Usually week 1 or less.

Milestone B First usable API 0%

Goal

Make story ingestion/query routes usable without any database yet.

Estimated total

4-6 hours.

What you will learn

  • Pydantic models and simple API contract design.
  • Basic error handling and local text ingestion flow.

Checklist

Definition of done

Done when you can ingest a text file and fetch that story back through the API.

Loose pacing

Often week 1-2 combined with Milestone A.

Milestone C Retrieval MVP (first LLM answer early) 0%

Goal

Ship a real /ask with citations earlier than the old plan did.

Estimated total

6-8 hours.

What you will learn

  • Chunking strategy and stable chunk IDs as citations.
  • Keyword retrieval plus a first grounded prompt with "don't know" behavior.

Checklist

You will need

/search needs no external account. /ask does need an LLM connection such as OpenAI, Anthropic, or a local Ollama model.

Definition of done

Done when /ask returns an answer with citations and refuses unsupported claims.

Loose pacing

Usually reaches first LLM-powered answer by week 2-4.

Milestone D Persistence + vector search 0%

Goal

Move from in-memory MVP to persistent storage and semantic retrieval.

Estimated total

10-14 hours.

What you will learn

  • Docker Compose + Postgres only when you need persistence.
  • How vector retrieval differs from keyword retrieval.
  • How embeddings are ingested as a batch step, not per request.

Checklist

You will need

Docker and Postgres start here, not earlier. You will also need an embeddings source, either a hosted API or a local model.

Definition of done

Done when vector retrieval works end-to-end and can be compared against keyword retrieval.

Loose pacing

This is where pace slows down due to infra and debugging overhead.

Milestone E Evaluation harness + traces 0%

Goal

Adopt a harness mindset: measure behavior, compare changes, and debug with evidence.

Estimated total

8-12 hours.

What you will learn

  • How to create a repeatable question set and baseline report.
  • How to evaluate retrieval separately from full answer behavior.
  • How request IDs and traces make failures debuggable.

Checklist

You will need

No extra SaaS is required for the first version. Start with local scripts and simple trace storage.

Definition of done

Done when you can run an eval script and inspect traces to debug failures.

Loose pacing

Later-phase effort: complexity rises because evaluation and observability are systems work.

Milestone F Demo + polish 0%

Goal

Turn the project into a clean, shareable demo and reusable template.

Estimated total

4-6 hours.

What you will learn

  • How to present the project so other people can run it quickly.
  • How to explain how the same project can work for other document sets.

Checklist

Definition of done

Done when a small demo UI makes the project easy to show to others.

Loose pacing

Usually the final 1-2 weeks. Keep provider switching secondary to demo usability.

What you can reuse after this project

  • The API contract and ingestion pattern for any plain-text corpus.
  • Chunk/citation discipline that transfers to PDFs, manuals, and internal docs.
  • Eval and trace workflow for continuous RAG quality improvement.

This roadmap starts with Lovecraft for focus and public-domain convenience, but the architecture is intentionally reusable for other document sets.
