Lovecraft RAG Engine — Learning roadmap (beginner-friendly)

A learning-first roadmap for building a Lovecraft RAG project from zero: Python basics, FastAPI, API design, retrieval, LLM integration, persistence, vector search, evaluation, tracing, and a small demo.

Shape: 12 learning phases Pacing: smaller conceptual jumps for beginners Output: reusable Lovecraft-to-any-docset RAG template
What you will build: A FastAPI RAG service over public-domain Lovecraft text: ingestion, chunking, keyword retrieval, answer generation with citations, then Postgres, pgvector, evals, traces, and a demo UI. What this teaches: Python and FastAPI basics first, then retrieval and secrets handling, then the more realistic engineering work around persistence, embeddings, evaluation, and debugging. How to use this roadmap: Work through phases in order. Each phase is smaller than the old milestone view, so the learning curve is smoother and blockers are easier to isolate.
Tip: early phases are intentionally small. Later phases slow down because persistence, embeddings, evaluation, and tracing are real systems work.
Phase 0 Python and environment basics 0%

Goal

Set up a clean Python environment before touching FastAPI.

Estimated total

1-2 hours.

What you will learn

  • What .venv is and why dependency isolation matters.
  • Why requirements.txt makes setup reproducible.

Checklist

Definition of done

Done when a clean shell can recreate your environment from requirements.txt.

You will need

Only Python and a terminal. No FastAPI knowledge required yet.

Phase 1 FastAPI startup basics 0%

Goal

Boot a minimal FastAPI app and understand what app.main:app means.

Estimated total

1-2 hours.

What you will learn

  • The difference between FastAPI and Uvicorn.
  • What a route is and how to verify one endpoint works.

Checklist

Definition of done

Done when /health runs locally and you can explain what app.main:app points to.

You will need

FastAPI and Uvicorn only. No stories or models yet.

Phase 2 Request and response thinking 0%

Goal

Learn request bodies, path parameters, status codes, and Pydantic models.

Estimated total

2-3 hours.

What you will learn

  • How schemas act as DTOs at the API boundary.
  • When to use 201, 400, 404, and 422.

Checklist

Definition of done

Done when one story can be created and fetched back with intentional status codes.

You will need

No DB and no RAG concepts yet. This is still FastAPI contract work.

Phase 3 Ingestion and local data handling 0%

Goal

Understand how local text becomes application data before introducing persistence.

Estimated total

2-3 hours.

What you will learn

  • Why in-memory storage is acceptable for early learning.
  • How to validate text ingestion cleanly.

Checklist

Definition of done

Done when stories can be ingested and listed without any database.

You will need

Only local text samples. Persistence still waits.

Phase 4 Retrieval without any LLM 0%

Goal

Build chunking and search before touching model integration.

Estimated total

3-4 hours.

What you will learn

  • Why chunking is the foundation of citations.
  • Why POST /search should be debugged before /ask.

Checklist

Definition of done

Done when POST /search returns sensible chunks for obvious questions without any LLM.

You will need

No secrets and no model account. This phase is retrieval-only.

Phase 5 Secrets and hosted model setup 0%

Goal

Learn how to handle API keys and model configuration safely before building /ask.

Estimated total

1-2 hours.

What you will learn

  • Why secrets belong in .env, not in route files.
  • The difference between a generation model and an embedding model.

Checklist

Definition of done

Done when the app can read model configuration safely and fail clearly if a key is missing.

You will need

One provider choice, such as OpenAI, Anthropic, or a local Ollama setup.

Phase 6 First grounded answer 0%

Goal

Build the first real POST /ask using retrieved context and citations.

Estimated total

3-4 hours.

What you will learn

  • Prompt construction and grounded behavior.
  • Why “I don’t know” is a quality feature, not a failure.

Checklist

Definition of done

Done when answerable questions return citations and unsupported questions refuse cleanly.

You will need

A working model connection from Phase 5 and retrieval from Phase 4.

Phase 7 Persistence 0%

Goal

Move from temporary in-memory state to persistent storage with Postgres.

Estimated total

3-5 hours.

What you will learn

  • Why persistence should come before vector retrieval.
  • How Docker Compose helps local dependency setup.

Checklist

Definition of done

Done when stories and chunks survive restarts and can be queried from the DB.

You will need

Docker, Postgres, and patience for a slower phase.

Phase 8 Embeddings and vector search 0%

Goal

Add semantic retrieval after persistence is stable.

Estimated total

4-6 hours.

What you will learn

  • What embeddings are and why they differ from generation models.
  • Why embeddings should be batch-ingested, not generated per request.

Checklist

Definition of done

Done when vector retrieval works and you can compare it against keyword retrieval on the same questions.

You will need

An embeddings source, hosted or local, plus the persistence work from Phase 7.

Phase 9 Evaluation 0%

Goal

Create a repeatable way to measure retrieval and answer quality.

Estimated total

3-4 hours.

What you will learn

  • The difference between retrieval metrics and answer metrics.
  • Why baseline reports matter before tuning.

Checklist

Definition of done

Done when you can run one eval command and compare results against a saved baseline.

You will need

A stable enough retrieval and answer pipeline to run repeatedly.

Phase 10 Traces and observability 0%

Goal

Make it possible to debug one failed request with evidence.

Estimated total

3-4 hours.

What you will learn

  • What request IDs and traces are for.
  • How to localize failures to retrieval, prompt construction, or generation.

Checklist

Definition of done

Done when one failed answer can be investigated without guessing.

You will need

The eval mindset from Phase 9 and a working answer pipeline.

Phase 11 Demo and product behavior 0%

Goal

Turn the backend into a shareable project that shows answers, citations, and uncertainty clearly.

Estimated total

3-5 hours.

What you will learn

  • Why AI UX includes citations and uncertainty handling.
  • How to present the project so it can be reused with other document sets.

Checklist

Definition of done

Done when a new person can run the demo and understand both support and uncertainty.

You will need

One UI tool choice and a stable enough backend to demo confidently.

Why this version is longer

The older 6-milestone version was good for project tracking, but the jumps were too steep for learning. This page is intentionally more granular so you can understand one concept at a time: FastAPI startup, schemas, ingestion, retrieval, secrets, model integration, persistence, vector search, evaluation, traces, and demo UX.

What you can reuse after this project

  • The API contract and ingestion pattern for any plain-text corpus.
  • Chunk/citation discipline that transfers to PDFs, manuals, and internal docs.
  • Secrets/config discipline for any app that calls external APIs.
  • Eval and trace workflow for continuous RAG quality improvement.

This roadmap starts with Lovecraft for focus and public-domain convenience, but the architecture is intentionally reusable for other document sets.
